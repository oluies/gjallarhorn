# Phase 1 Data Model: Conversation Wiring (Gjallarhorn side, Phase A companion)

**Feature**: 001-conversation-wiring
**Date**: 2026-05-23

Phase A's Gjallarhorn-side work introduces in-memory entities for the test harness and the integration test. No new on-wire or persistent state.

---

## E1. Harness (in-process service stack)

**Purpose**: The in-memory stand-up of every server-side component the Phase A integration test (and the Neverlur-side demo CLI) needs. Mirrors the production topology so the integration test exercises every real code path.

**Fields**:
- `NeverlurCoordinator *coordinator.Server` ‚Äî Neverlur coordinator (handles add-friend + dialing rounds)
- `NeverlurMixers []*mixnet.MixServer` ‚Äî 3 Neverlur mixers
- `NeverlurPKG *pkg.Server` ‚Äî Neverlur PKG (handles identity attestation)
- `NeverlurCDN *cdn.Server` ‚Äî Neverlur CDN (mailbox storage for friend-discovery)
- `GjallarhornCoordinator *coordinator.Server` ‚Äî distinct from Neverlur's; handles convo rounds
- `GjallarhornMixers []*mixnet.MixServer` ‚Äî 3 Gjallarhorn mixers
- `GjallarhornCDN *cdn.Server` ‚Äî distinct from Neverlur's
- `AddFriendConfig *config.SignedConfig` ‚Äî bootstrap config for add-friend service (v2 hybrid)
- `DialingConfig *config.SignedConfig` ‚Äî bootstrap config for dialing service (v2 hybrid)
- `ConvoConfig *config.SignedConfig` ‚Äî bootstrap config for convo service (v2 hybrid)
- `HarnessGuardian *hybrid.HybridIdentity` ‚Äî single-guardian identity used to sign all three configs at harness `New()` time
- `ListenAddr string` ‚Äî Unix socket path for the Neverlur-side demo CLI to connect to (empty in pure-test mode)
- `tempDir string` ‚Äî `t.TempDir()` for any on-disk state (PKG database, etc.)

**Lifecycle**:
- `New(tb testing.TB) *Harness` ‚Äî constructs every component, generates the harness guardian, signs the three configs, starts the goroutines that drive each server, registers `tb.Cleanup` for teardown.
- `Close()` ‚Äî explicit teardown (defer-friendly).
- `AdvanceRound(service string)` ‚Äî manually moves the named service ("AddFriend", "Dialing", "Convo") forward one round, bypassing the real clock. Lets tests be deterministic.

**Validation**:
- `New` panics (`tb.Fatal`) if any component fails to start.
- `AdvanceRound("UnknownService")` is a test-author error and panics.

## E2. Test Client (per-username Neverlur client wired against a Harness)

**Purpose**: A Neverlur client configured to talk to a specific harness instance. Each integration test typically creates two (Alice + Bob) plus possibly cover-traffic clients.

**Fields**:
- `*neverlur.Client` ‚Äî the standard client, embedded
- `Username string` ‚Äî the registered username
- `HybridIdentity *hybrid.HybridIdentity` ‚Äî generated by `hybrid.GenerateHybridIdentity()` at `ClientFor()` time
- `RecvCh chan IncomingMessage` ‚Äî buffered channel that captures everything Bob (or Alice) receives; tests read from this to assert delivery
- `ConvoState *gjallarhorn.Conversation` ‚Äî populated by the `SendingCall`/`ReceivedCall` callbacks; nil until the call bootstraps

**Lifecycle**:
- `Harness.ClientFor(tb testing.TB, username string) *TestClient` ‚Äî generates identity, registers with the harness's PKG, creates a `neverlur.Client` with handler callbacks wired to `RecvCh`, returns ready-to-use.
- Connections to the harness's coordinators stay open for the life of the test (cleaned up by `Harness.Close()`).

**Validation**:
- `ClientFor` fails the test if registration fails.

---

## E3. IncomingMessage (test fixture)

**Purpose**: One element of `TestClient.RecvCh`. Captures everything a single conversation packet delivery produces so the test can assert against it.

**Fields**:
- `FromPeer string` ‚Äî the peer's username
- `Body []byte` ‚Äî the decrypted message bytes
- `Round uint32` ‚Äî the convo round in which it was delivered
- `Err error` ‚Äî non-nil if delivery failed (e.g. decryption errored)

**Lifecycle**:
- Created by the test client's receive callback (wired via Gjallarhorn's `Conversation.Open` returning value).
- Lives only inside the test's `RecvCh`.

---

## E4. Harness Guardian (single-guardian setup for test configs)

**Purpose**: The single hybrid identity that signs the three SignedConfig records the harness produces at startup. In production these come from multiple independent guardians; for tests one is enough.

**Fields**:
- `*hybrid.HybridIdentity` ‚Äî the standard 3-primitive identity (Ed25519 + ML-DSA-65 + ML-KEM-768)

**Lifecycle**:
- Generated by `hybrid.GenerateHybridIdentity()` in `Harness.New()`.
- Used to sign three configs; each config lists this identity as its sole Guardian.
- Discarded at `Harness.Close()` (no persistence).

**Validation**:
- `VerifyBinding()` is called immediately after generation; harness `New()` panics if it fails.

---

## E5. Forbidden-Pattern Match (static-check test fixture)

**Purpose**: A struct representing a single hit of one of the forbidden patterns in `TestNoClassicalSessionKeySource`. Used to produce structured diagnostics.

**Fields**:
- `Pattern string` ‚Äî e.g. `"box.Precompute"`
- `File string` ‚Äî relative path from the repo root
- `Line int` ‚Äî line number
- `Snippet string` ‚Äî the matched source line

**Lifecycle**:
- Produced by the AST walk in the test.
- Aggregated into a per-test slice; if non-empty after the walk completes, the test fails with one diagnostic per hit.

## What this commit does NOT add

- No production-code data structures.
- No new on-wire formats.
- No new on-disk file formats.
- No new persistent storage layouts in any server-side component.

Every entity here lives only in test goroutines and is garbage-collected at test end.
